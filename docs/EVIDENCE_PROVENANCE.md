# Evidence Provenance

x-harness treats completion cards, command output, and agent-written evidence as
untrusted input. Evidence is stronger when it is generated by a runner, bound to
repository state, and protected by hashes or external provenance.

This document describes the recommended evidence hierarchy and the fields that
strict verification can enforce. It does not turn x-harness into a correctness
oracle: strong provenance shows where evidence came from, not that the product
behavior is semantically correct.

## Recommended Evidence Hierarchy

Prefer evidence in this order:

1. CI evidence produced by a trusted workflow and linked with `ci_run_url`,
   artifact path, and artifact checksum.
2. Local evidence produced by `xh evidence run -- <command>`.
3. Manually recorded command evidence with command, exit code, runner,
   timestamps, and output or artifact hashes.
4. Free-form manual rationale. Use this only for `light` tier or when command
   execution is impossible and the card declares the limitation.

## Capture Command Evidence

Use `xh evidence run` when recording local command-backed evidence:

```bash
xh evidence run -- npm test
xh evidence run -- go test ./...
```

By default, the command writes a deterministic JSON record under
`.x-harness/evidence/`. The record includes:

- command and arguments;
- working directory;
- exit code;
- start and end timestamps;
- duration;
- stdout/stderr byte counts and SHA-256 hashes;
- bounded stdout/stderr captured prefixes;
- git root, commit, branch, dirty flag, and dirty baseline hash when available.

Completion cards can then reference the command fields directly in
`evidence.command_evidence[]` or attach the record as a verification artifact.

## CI Artifact Binding

For CI-produced evidence, include enough data to reconnect the card to the
actual workflow run:

- `runner`, for example `github-actions-node-22`;
- `command`;
- `exit_code`;
- `started_at` and, when available, `ended_at`;
- `ci_run_url`;
- `artifact_path` for persisted reports, logs, coverage, SBOMs, or release
  files;
- `artifact_hash` for persisted artifacts;
- `stdout_hash` or `stderr_hash` when output is not persisted separately;
- repository commit or a dirty-state baseline hash if the run was not on a clean
  commit.

The strongest release evidence also includes release checksums, SBOMs, and
Sigstore bundles generated by the release workflow.

## Runtime Enforcement

`xh verify --strict` requires `standard` and `deep` command-backed evidence to
include:

- `command`;
- `exit_code`;
- `runner`;
- `started_at`.

`xh verify --require-evidence-hash` additionally requires `standard` and `deep`
`command_evidence[]` and `verification_artifacts[]` records to include at least
one of:

- `stdout_hash`;
- `stderr_hash`;
- `artifact_hash`.

The `governed-deep` profile enables the evidence-hash requirement by default.
Other profiles can opt in with `--require-evidence-hash`.

## Optional Signing And Attestation

For release and high-risk workflows, prefer external provenance in addition to
card fields:

- GitHub Actions OIDC-backed npm provenance for package publication;
- Sigstore or cosign bundles for release binaries;
- signed checksums for release artifacts;
- SBOMs tied to the same release commit and checksums.

These signals are not a substitute for tests or review. They make artifact
origin and tampering easier to audit.

## Limits

Evidence provenance does not prove that:

- the tests cover the intended behavior;
- the command was the right command for the task;
- a compromised local machine or CI runner told the truth;
- source code outside the tested scope is correct or secure.

x-harness uses provenance to make completion claims structured, auditable, and
policy-controlled. It should be paired with CI, review, security checks, and
task-specific acceptance tests.
