# Threat Model

This document defines what x-harness is designed to protect, what it assumes,
and where its guarantees stop. It is intentionally conservative: x-harness is a
verification harness for completion claims, not a software correctness oracle.

## Assets To Protect

- Admission decisions: `admission.outcome: success` and
  `acceptance_status: accepted` must mean the completion card satisfied the
  repository policy.
- Completion evidence: command evidence, verification artifacts, trace events,
  packets, and release artifacts should remain attributable and tamper-evident
  where the current feature set supports it.
- Repository contract: schemas, policies, templates, adapters, and managed
  context blocks should not silently drift from the runtime rules.
- Verifier read-only boundary: `xh verify` must not repair or mutate the work
  product while deciding whether a card is accepted.
- Recovery routing: withheld outcomes should keep owner and next-action
  information instead of collapsing into an ambiguous failure.

## Trust Boundaries

- **Untrusted**: source code changed by an agent, completion cards, sub-agent
  returns, command output pasted by an agent, logs, evidence files, adapter
  text, and user-provided artifacts.
- **Trusted within a local run**: the x-harness binary/source currently being
  executed, the repository policy files selected for that run, and the local
  filesystem view used by the verifier.
- **Conditionally trusted**: CI artifacts, command evidence records, packet
  chains, trace chains, and release artifacts. They are stronger when bound to
  hashes, exit codes, timestamps, repository state, checksums, signatures, or CI
  provenance.
- **Out of process**: agent platforms, LLM providers, editor plugins, shell
  history, and external CI systems. x-harness can consume artifacts from them,
  but it does not control them.

## Attacker And Failure Capabilities

x-harness is strongest against accidental and procedural failures:

- An agent declares completion too early.
- Required evidence is missing or structurally invalid.
- A completion card contradicts itself, for example success with withheld
  status.
- A verifier path unexpectedly mutates files while verification is running.
- Adapter or managed context files drift from canonical rules.
- A high-risk command or protected path lacks required approval metadata.

x-harness has limited protection against intentional deception unless stronger
provenance is used:

- An agent may fabricate command evidence inside a completion card.
- An agent may paste modified command output.
- A compromised runner may execute commands in a different environment than the
  card claims.
- A malicious actor with write access to repository policy can weaken the
  policy before running verification.
- A compromised local machine can tamper with both evidence and verifier input.

## Evidence Trust Rules

Prefer evidence generated by `xh evidence run` or CI automation over free-form
agent text. Strong evidence should include:

- command and arguments;
- working directory;
- exit code;
- start/end timestamps;
- stdout/stderr hashes or persisted artifact hashes;
- repository commit, branch, dirty-state flag, and dirty baseline hash when
  available;
- CI run URL or release artifact checksum when produced by automation.

For standard and deep cards, strict verification requires command-backed
evidence to include command, exit code, runner, and start time. The
`--require-evidence-hash` flag and the `governed-deep` profile additionally
require at least one output or artifact hash. These requirements improve
provenance, but they do not prove that tests cover the intended behavior or that
the software is bug-free.

## Evidence That Should Be Re-run

Re-run evidence when:

- the command was only described by an agent and not generated by
  `xh evidence run` or CI;
- the card references outputs without hashes or artifacts;
- the repository changed after evidence was captured;
- the task touches high-risk, protected, release, security, or migration paths;
- the runner environment differs from the expected development or CI
  environment;
- review depends on semantic correctness rather than contract consistency.

## What x-harness Does Not Protect Against

x-harness does not:

- guarantee that generated code is correct, secure, or production-ready;
- replace tests, CI, human review, or security review;
- prove that a test suite covers the bug being fixed;
- sandbox arbitrary commands;
- prevent a compromised machine from lying about local state;
- prevent a repository maintainer from weakening policy;
- verify external agent-platform behavior beyond files and artifacts available
  to the local run;
- grant PGV, prediction, approval-risk, or attribution outputs admission
  authority by default.

## Recommended Security Posture

- Treat completion cards and logs as untrusted input.
- Use `xh verify --strict` or a strict verify profile in CI.
- Prefer `xh evidence run -- <command>` for local command evidence.
- Bind CI evidence to run URLs, checksums, and artifacts.
- Require approval records for high-risk commands and protected paths.
- Keep schemas, policies, templates, adapters, and docs under normal code
  review.
- Use release checksums, SBOMs, and Sigstore bundles when consuming release
  artifacts.
